# Design Decisions and Trade-offs

## Decision log

| Decision | Options considered | Day 1 choice | Why the others lost | Cost accepted | Add later |
|---|---|---|---|---|---|
| Client resilience and workflow freshness | Online-only web; mounted session with no-store XML; web with signed IndexedDB cache and outbox; native Android with SQLite | The form runs locally in the mounted React page after one XML fetch. The AI step needs a connection, workflow responses use `no-store`, and save retries reuse one in-memory submission key. | Online-only ignores the field conditions. IndexedDB needs outbox, conflict and retirement rules. An indefinite cache or silent upgrade can run unsafe rules or erase the audit trail. Android changes the delivery platform and duplicates the backend-AI boundary. | A full outage stops at AI; refresh loses the draft and key; a version bump blocks the encounter; a same-label XML edit is undetectable. | Add IndexedDB draft and outbox with a signed, content-addressed workflow manifest. Consider Android only if field evidence justifies a native client. |
| Outcome storage | Process memory; SQLite; Postgres | SQLite through `node:sqlite`, stored on the `screening-data` Docker volume. Tests use SQLite in memory. | Memory does not persist outcomes. Postgres adds another service and migration surface before the demo needs concurrent writers. | SQLite is a single-writer file tied to one API instance. | Postgres when the API needs replicas or concurrent writers. |
| AI boundary | Mock in Node; separate AI service; on-device model | A replaceable mock module sits behind `POST /api/v1/ai/evaluations`. | A separate service adds a network hop and operational work for a deterministic mock. An on-device model changes the required backend seam and raises browser bundle and hardware costs. | The mock cannot deploy or scale independently. | Move a real model behind the same API contract if its runtime or scaling needs differ. |
| Model release binding | Float any compatible model; pin an exact artifact; resolve a compatibility constraint at publication | Day 1 accepts `^MAJOR` because each bundled mock has one installed implementation. A compact descriptor validates its contract, and the resolved version is stored. | Floating a production model can change outcomes without clinical review. Pinning inside every draft makes authoring and controlled patch upgrades awkward. | The descriptor lives in application code and is not independently versioned or signed with the workflow. | Publication resolves the constraint against a versioned descriptor, tests it, records the artifact digest and requires clinical reapproval for a changed model. |
| Workflow language | Executable expressions; constrained XML; screening-specific React code | Engine 1 uses typed steps, fixed operators, explicit links and allow-listed field types. The server validates before serving. | Expressions enlarge the injection and policy surface. Screening-specific React code breaks the central configuration-only requirement. | A new step type needs an engine release. Authors cannot add arbitrary calculations. | A reviewed engine version with migration guidance and compatibility tests. |
| Submission idempotency | Server-issued key; `encounterId`; client submission UUID; last write wins | The browser creates a submission UUID. The server stores the UUID, canonical payload hash and original response with the outcome. Same hash replays; changed hash returns `409`. | A server-issued key needs an extra successful request before work can be retried. `encounterId` couples transport identity to a clinical record. Last-write-wins can hide a mutated retry. | The guarantee lasts only while the mounted encounter retains the key. Refresh, close, crash or device switch can lose it. Replay records must live as long as outcomes. | IndexedDB persistence, authenticated cross-device encounter ownership and a written amend-and-resubmit policy. |
| Image handling | Base64 JSON; multipart attachment; direct object-store upload | XML binds the image with `bind="file"`. The evaluation endpoint accepts the bound JPEG or PNG, records hash and size, and does not store the blob. The measured value drives this mock. | Base64 inflates a weak-network request. Object storage adds credentials, retention and cleanup to a demo that does not need to retrieve the image. | The evidence image cannot be reviewed later, and the Day 1 mock adapter does not consume its pixels. | A model descriptor distinguishes evidence from inference input. Add object storage, signed upload, content inspection, retention and image-quality guidance when a real use case needs the file. |
| Workflow authoring | XML in Git; upload endpoint; authoring portal | Two XML files, a validation command and server validation. No upload endpoint or RBAC. | An upload route without review and publishing controls is unsafe. A full portal is a separate product and the brief asks only for its design. | Clinical authors still need help from the repository workflow. | Draft, review, publish, retire, rollback, audit history and author/reviewer RBAC. |

## Trade-off register

| Tension | Call | Cost |
|---|---|---|
| Expressiveness vs safety | Use a small XML language with typed elements and structured conditions. No scripts, HTML or expression strings. | Some future screening rules will need a new engine version instead of XML alone. |
| Compatibility vs continuity | Freeze the Engine 1 syntax and fail closed when an old client meets new syntax. Additive API response fields stay in `v1`; breaking meaning or request changes use `v2`. | A stale client may block a screening instead of degrading. Supporting old engines and API versions creates a retirement burden. |
| Offline use vs workflow freshness | Require connectivity for the AI step and fetch the active XML without caching. | A full outage stops the screening. This avoids running an urgently retired workflow from a stale cache. |
| Client flow vs server trust | The browser renders and branches for a responsive flow. The server reloads the workflow, validates answers, trusts only stored AI evaluations and recomputes the final outcome. | Logic runs on both sides. The shared engine reduces drift but makes its tests important. |
| Idempotency vs genuine re-screen | Use a submission UUID separate from the encounter. Reuse it for an exact retry; create a new encounter and key for a real re-screen. | The guarantee ends when the mounted page loses its key. Refresh and cross-device duplicates are not detected. |
| Generic renderer vs humane UX | Use one allow-listed component registry with large targets, one question step at a time and explicit wait, error and retry states. | XML cannot invent a bespoke layout. A genuinely new interaction needs a reviewed component and engine release. |
| Authorability vs rigor | Use readable XML and plain condition elements, then reject invalid graphs and unsupported types. | Raw XML is still not a good long-term tool for clinical staff. The authoring product is deferred. |
